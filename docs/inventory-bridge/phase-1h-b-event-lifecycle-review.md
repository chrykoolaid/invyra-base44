# Phase 1H-B Bridge Event Lifecycle Review

Status: review only  
Component: Inventory / `chrykoolaid/invyra-base44`  
Runtime state: not implemented and not activated

## Purpose

Phase 1H-B reviews the future event lifecycle for the ScanOps <-> Inventory Bridge from the Inventory side.

This is documentation only. It does not implement runtime bridge code, transport, sync, ingestion, replay, entities, persistence writes, Inventory writes, or operational mutation.

## Lifecycle overview

The future bridge event lifecycle should remain evidence-first and Inventory-controlled:

```text
Captured in ScanOps
  -> queued by ScanOps
  -> future transport attempt
  -> received by Inventory trust gate
  -> classified by Inventory
  -> accepted to ledger, rejected, duplicated, quarantined, or deferred
  -> receipt generated by Inventory
  -> ScanOps reconciles receipt
  -> optional later Inventory workflow review
```

## Inventory lifecycle states

Inventory should own future classification states such as:

```text
RECEIVED
ACCEPTED_TO_LEDGER
REJECTED_SCHEMA
REJECTED_TRUST
REJECTED_SCOPE
DUPLICATE_EVENT
QUARANTINED
TEMPORARY_FAILURE
OPERATOR_REVIEW_REQUIRED
ARCHIVED
```

These states classify evidence. They do not directly create stock, price, POS, order, forecast, or Item Master changes.

## Inventory lifecycle responsibilities

Inventory is responsible for future decisions around:

- Whether the event belongs to the correct Inventory instance.
- Whether the source device is trusted.
- Whether the schema version is supported.
- Whether the event type is allowed.
- Whether the payload is intact.
- Whether the event is duplicate or replayed.
- Whether the event should be quarantined.
- Whether operator review is required.
- Whether a receipt should allow retry.
- Whether the bridge is disabled by stop controls.

## Transition review

| Transition | Inventory review requirement | Operational mutation allowed |
| --- | --- | --- |
| Received -> Accepted to ledger | Trust, schema, scope, idempotency, payload checks | No |
| Received -> Rejected | Clear reason code | No |
| Received -> Duplicate | Existing event or idempotency key match | No |
| Received -> Quarantined | Suspicious, malformed, stale, or review-required evidence | No |
| Accepted to ledger -> Operator review | Explicit workflow boundary | No direct mutation |
| Operator review -> Operational action | Future separate workflow only | Not part of bridge lifecycle |
| Any state -> Archived | Retention policy only | No |

## Receipt lifecycle

Inventory should generate a receipt for every future submission classification.

Receipt outcomes should include:

```text
ACCEPTED_TO_LEDGER
REJECTED_SCHEMA
REJECTED_TRUST
REJECTED_SCOPE
DUPLICATE_EVENT
QUARANTINED
TEMPORARY_FAILURE
```

A receipt must not claim operational mutation unless a later separate Inventory workflow actually performs that mutation.

## Archive boundary

Archiving a bridge event should be a retention and audit action only.

Archiving must not create or reverse operational Inventory records.

## No-mutation lifecycle rule

The bridge event lifecycle must not directly mutate:

- Stock movements.
- Item Master records.
- Price records.
- POS sale records.
- Order records.
- Forecast records.
- Posted wastage records.
- Posted store-use records.
- Markdown price activation records.

## Future implementation questions

Before implementation, the team must answer:

- Which states are immutable?
- Which metadata fields can be updated after classification?
- Who can archive events?
- How long should evidence be retained?
- How are duplicate receipts linked to original events?
- How are quarantine events surfaced?
- How is operator review separated from bridge receipt classification?
- How are archived records audited?
